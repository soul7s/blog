## 준비물

시작하기 전에 준비한 것은 아래와 같다.

- 코덱스 같은 AI 도구
- GitHub
- Vercel
- 원하는 디자인을 고를 수 있는 [https://getdesign.md/](https://getdesign.md/)

## 전체 흐름 한눈에 보기

1. [https://getdesign.md/](https://getdesign.md/) 에서 원하는 디자인을 고른다.
2. `DESIGN.md`를 복사해서 코덱스에게 블로그 제작을 요청한다.
3. 결과물을 테스트해보고 마음에 안 드는 부분을 수정한다.
4. GitHub에 새 저장소를 만들고 프로젝트를 업로드한다.
5. Vercel에서 GitHub 저장소를 연결해 배포한다.
6. 이후에는 원하는 글을 요청해서 작성하거나 직접 수정한 뒤 GitHub에 반영한다.
7. 필요하면 오픈클로 같은 도구와 연결해서 작성부터 등록까지 더 자동화한다.

## 1. 원하는 디자인 먼저 고르기

먼저 [https://getdesign.md/](https://getdesign.md/) 에서 내가 원하는 분위기의 디자인을 골랐다.

여기서 한 일은 단순하다.

1. 원하는 디자인 페이지로 들어간다.
2. `DESIGN.md`를 누른다.
3. 복사 버튼을 눌러 내용을 가져온다.

이 단계가 중요한 이유는, AI에게 "예쁜 블로그 만들어줘"라고 말하는 것보다 어떤 구조와 스타일을 원하는지 구체적으로 전달할 수 있기 때문이다. 말로 설명하는 것보다 이미 정리된 디자인 문서를 같이 주는 편이 훨씬 정확하다.

## 2. 코덱스에게 블로그 제작 요청하기

디자인을 정한 뒤에는 코덱스에게 아래처럼 요청했다.

```text
Next.js 기반의 정적 블로그 툴을 만들어보려고해.
- 마크다운으로 내 프로젝트 폴더에서 글을 관리할수있어야해.
- 각 블로그 아티클은 주제별로 폴더를 다르게 저장할수있어야해.
- 아티클 파일 제목은 반드시 날짜-제목.md 같은 형태여야해.
- 이거는 나중에 깃허브에 푸시를 할거야. 그리고 그것을 버셀과 연동하려고해.
- 마크다운으로 작성한 글이 블로그처럼 렌더링되는 기능이 필요해
- 디자인은 아래 링크와 동일하게 구성해줘
```

여기서 마지막 줄의 "아래 링크"에는 내가 고른 디자인 자료나 [https://getdesign.md/](https://getdesign.md/) 에서 복사한 `DESIGN.md` 내용을 함께 넣었다.

## 3. 이 요청이 실제로 뜻하는 것

위 요청을 쉽게 풀면, 그냥 블로그 하나를 만들어달라는 뜻이 아니었다. 아래 조건을 모두 만족하는 구조를 원한 것이다.

- `Next.js` 기반으로 돌아가는 정적 블로그일 것
- 글은 관리자 페이지가 아니라 프로젝트 폴더 안의 Markdown 파일로 관리할 것
- 글은 주제별 폴더로 나눠서 저장할 수 있을 것
- 파일명은 `날짜-제목.md` 형식으로 통일할 것
- 나중에 GitHub에 올리고 Vercel과 연결해도 문제없이 배포될 것
- Markdown으로 쓴 글이 실제 블로그 페이지처럼 렌더링될 것
- 내가 고른 디자인 분위기를 최대한 그대로 반영할 것

즉, 핵심은 "글 작성은 단순하게, 정리는 깔끔하게, 배포는 쉽게"였다.

## 4. 결과물을 테스트하고 수정하기

초안이 완성되면 바로 끝내지 않았다. 먼저 테스트를 한 번 돌려보고, 실제 화면에서 어색한 부분을 확인한 뒤 다시 수정했다.

이 과정에서 주로 보게 되는 것은 아래와 같다.

- 메인 목록이 읽기 편한지
- 글 상세 페이지가 너무 답답하거나 허전하지 않은지
- 폰트, 여백, 카드 스타일이 내가 원한 느낌과 맞는지
- Markdown으로 쓴 글이 깨지지 않고 잘 렌더링되는지

처음 결과물은 대체로 방향만 맞는 경우가 많아서, 이 수정 단계가 생각보다 중요하다.

## 5. GitHub에 새 저장소를 만들고 업로드하기

블로그가 어느 정도 완성되면 GitHub에서 새 저장소를 만든다. 그다음 저장소 주소를 준비한 뒤, 코덱스에게 업로드를 요청했다.

이렇게 하면 로컬에서만 있던 프로젝트가 GitHub에 정리되고, 이후 수정 이력 관리도 쉬워진다. 나중에 다른 기기에서 작업할 때도 훨씬 편하다.

## 6. Vercel에 연결해서 배포하기

GitHub 업로드가 끝나면 Vercel로 가서 새 프로젝트를 추가한다. 그다음 GitHub 저장소를 가져오는 방식으로 연결한 뒤 배포하면 된다.

이 부분은 한 번만 설정해두면 이후가 정말 편하다. 나중에는 GitHub에 변경사항만 반영해도 배포까지 거의 자동으로 이어진다.

전체 흐름은 아래처럼 보면 된다.

1. Vercel에서 새 프로젝트를 만든다.
2. GitHub 저장소를 가져온다.
3. 빌드 설정을 확인한다.
4. 배포한다.

## 7. 이후 글 작성과 수정 방식

배포가 끝난 뒤에는 내가 원하는 내용을 요청해서 글을 작성하면 된다. 수정이 필요하면 직접 Markdown 파일을 고쳐도 되고, 다시 요청해서 수정한 뒤 GitHub에 반영해도 된다.

중요한 점은 GitHub와 Vercel이 연결되어 있다는 것이다. 그래서 수정본을 GitHub에 등록하기만 하면, 다시 배포하는 과정이 훨씬 단순해진다.

## 8. 내가 추가로 연결한 방식

나는 이 흐름을 더 편하게 쓰려고 오픈클로도 연결했다. GitHub 권한을 연결해두었기 때문에, 텔레그램으로 요청하면 글 작성부터 등록까지 바로 이어지게 만들었다.

이렇게 해두면 꼭 컴퓨터 앞에 앉아 있지 않아도 초안 작성이나 등록 작업을 빠르게 진행할 수 있다.

## 실수하지 않으려고 남기는 메모

다음에 다시 만들 때는 아래만 꼭 기억하면 된다.

- 디자인은 먼저 정하고 시작할 것
- `DESIGN.md`를 복사해서 AI에게 최대한 구체적으로 전달할 것
- 파일 구조와 파일명 규칙은 처음부터 정해둘 것
- GitHub 업로드 전에 테스트를 한 번 해볼 것
- Vercel 연결은 GitHub 저장소가 잘 올라간 뒤에 진행할 것

## 마무리

정리하면, 이 블로그를 만든 흐름은 "디자인 선택 -> AI에게 구조 요청 -> 테스트 및 수정 -> GitHub 업로드 -> Vercel 배포 -> 이후 작성 자동화" 순서였다.

나중에 다시 처음부터 만들어야 할 일이 생기면, 이 글을 그대로 따라가면 된다. 적어도 어디서부터 시작해야 할지 몰라서 막히는 일은 줄어들 것이다.

오픈클로(OpenClaw)를 처음 보게 되면 보통 설치 명령어부터 찾게 된다. 그런데 실제로는 설치보다 먼저 헷갈리는 부분이 따로 있다. **어디가 공식 홈페이지인지, 어디가 문서인지, 내가 실제로 접속해서 설정을 바꾸는 화면은 어디인지**가 처음부터 분리되어 있지 않으면 설정 과정이 꽤 혼란스러워진다. 거기에 텔레그램 연결까지 붙이기 시작하면 "이건 홈페이지에서 하는 건가, 터미널에서 하는 건가, 봇파더에서 하는 건가"가 한꺼번에 섞이기 쉽다.

이번 글은 그런 혼란을 줄이기 위해 정리한 기록이다. 오픈클로를 처음 붙일 때 알아두면 좋은 **주소의 역할**, **모델을 왜 코덱스로 골랐는지**, **홈페이지처럼 보이는 화면에서 무엇을 설정하면 되는지**, 그리고 **텔레그램으로 어떻게 연결하는지**를 순서대로 정리해본다.

## 1. 먼저 구분해야 하는 세 가지 주소

오픈클로를 처음 만질 때 가장 먼저 정리해야 하는 건 주소다. 비슷해 보이지만 역할은 서로 다르다.

### 1-1. 공식 홈페이지

- `https://openclaw.ai`

이 주소는 오픈클로를 처음 소개받을 때 보는 공식 사이트에 가깝다. 프로젝트 소개, 제품 방향, 전체 느낌을 파악할 때 보는 입구라고 생각하면 된다.

### 1-2. 공식 문서

- `https://docs.openclaw.ai`

실제 설정 방법, 채널 연결, OAuth, 텔레그램, 보안, 컨트롤 UI 같은 세부 설명은 대부분 문서 사이트에 있다. 오픈클로는 기능 폭이 넓어서, 막히기 시작하면 결국 이 문서로 돌아오게 된다.

### 1-3. 내가 실제로 접속해서 쓰는 로컬 컨트롤 UI

- 기본 주소: `http://127.0.0.1:18789/`
- 또는 `http://localhost:18789/`

이게 흔히 말하는 "홈페이지처럼 보이는 화면"에 더 가깝다. 오픈클로 문서에서는 이 화면을 **Control UI**라고 부른다. 브라우저에서 열리는 관리 화면이고, 게이트웨이가 떠 있어야 접속된다. 즉 공식 홈페이지가 아니라, **내가 띄운 오픈클로 인스턴스의 로컬 웹 화면**이다.

이 차이를 먼저 알아두면 혼란이 많이 줄어든다.

- `openclaw.ai`는 소개 페이지
- `docs.openclaw.ai`는 설명서
- `127.0.0.1:18789`는 내가 실제로 만지는 관리 화면

처음에는 이 셋이 전부 비슷한 "홈페이지"처럼 느껴지지만, 역할은 완전히 다르다.

## 2. 오픈클로는 무엇을 만드는 도구인가

오픈클로는 단순한 채팅 앱이 아니라 **메신저와 AI 에이전트를 이어주는 셀프 호스팅 게이트웨이**에 가깝다. 공식 설명도 이 방향이다. 하나의 게이트웨이를 띄워두고, 그 위에 텔레그램 같은 채널을 연결하고, 모델을 붙이고, 세션과 메모리와 툴 사용을 관리하는 구조다.

중요한 건 이것이 "웹사이트 하나 만드는 것"이 아니라는 점이다. 실제로는 다음 세 가지를 동시에 세팅하는 과정에 가깝다.

1. 게이트웨이 실행
2. 모델 인증 및 기본 모델 선택
3. 채널 연결, 예를 들면 텔레그램 봇 연결

그래서 오픈클로를 세팅할 때는 UI보다 **런타임 구조**를 먼저 이해하는 편이 좋다. 브라우저 화면은 관리 도구일 뿐이고, 실제 중심은 게이트웨이다.

## 3. 처음 설치할 때는 `openclaw onboard` 기준으로 보는 게 제일 쉽다

공식 문서에서도 CLI onboarding을 권장한다. 가장 짧게 시작하려면 보통 아래 흐름으로 생각하면 된다.

```bash
npm install -g openclaw@latest
openclaw onboard --install-daemon
```

문서 기준으로 onboarding은 다음을 한 번에 잡아주는 역할을 한다.

- 모델과 인증 방식 선택
- 워크스페이스 위치 설정
- 게이트웨이 포트와 인증 설정
- 채널 연결
- 필요하면 데몬 설치

즉 처음부터 수동으로 이것저것 조각내기보다, **한 번은 onboard로 큰 틀을 잡고 들어가는 게 훨씬 편하다.**

오픈클로를 설치한다고 해서 바로 텔레그램이 붙는 건 아니다. 반대로 텔레그램 봇 토큰만 있다고 바로 원하는 AI 비서가 되는 것도 아니다. onboarding은 이 중간 과정을 한 번에 정리해주는 도구라고 보면 된다.

## 4. 모델은 다양하지만, 이번 설정에서는 왜 코덱스를 골랐는가

오픈클로는 한 모델만 쓰는 도구가 아니다. 여러 제공자와 모델을 붙일 수 있고, 실제 문서도 API 키 방식과 OAuth 방식 둘 다 설명하고 있다.

그중 이번 설정에서 코덱스 쪽을 고른 이유는 단순하다. **개인용으로 시작할 때 진입이 상대적으로 편했기 때문**이다.

정확히 말하면 오픈클로는 OpenAI 쪽에서 두 가지 길을 나눠서 본다.

1. `openai/*`
   - OpenAI API 키 기반
   - 사용량 과금형
2. `openai-codex/*`
   - ChatGPT/Codex OAuth 로그인 기반
   - 별도 API 키를 바로 들고 시작하지 않아도 됨

이 차이를 잘못 이해하면 "코덱스는 API가 없다"고 느낄 수 있는데, 더 정확한 표현은 이렇다.

**이번 세팅에서는 OpenAI API 키를 따로 발급받아 과금형으로 시작하기보다, ChatGPT/Codex 계정 로그인 기반의 OAuth 경로를 선택했다.**

이 방식의 장점은 분명하다.

- 이미 계정이 있는 사람에게 진입이 쉽다
- 개인 테스트를 빨리 시작하기 좋다
- 처음부터 API 키 발급, 사용량 관리, 과금 감각을 동시에 신경 쓰지 않아도 된다

그래서 "모델이 다양하지만 왜 코덱스로 갔나"에 대한 대답은, 성능 비교 하나만의 문제가 아니라 **초기 연결 난도와 계정 기반 시작의 편의성**에 더 가깝다.

## 5. 코덱스 기준 기본 모델을 잡는 흐름

문서 기준으로 코덱스 OAuth 경로는 대략 이런 흐름이다.

```bash
openclaw onboard --auth-choice openai-codex
```

또는 직접 로그인 흐름만 먼저 돌릴 수도 있다.

```bash
openclaw models auth login --provider openai-codex
```

그 다음 기본 모델을 이렇게 잡을 수 있다.

```bash
openclaw config set agents.defaults.model.primary openai-codex/gpt-5.4
```

여기서 중요한 건 이름이 비슷하다고 `openai/gpt-5.4`와 `openai-codex/gpt-5.4`를 같은 것으로 생각하면 안 된다는 점이다.

- `openai/...`는 API 키 경로
- `openai-codex/...`는 ChatGPT/Codex OAuth 경로

처음 세팅하는 입장에서는 이 두 경로를 섞지 않는 것이 가장 중요하다.

## 6. 사람들이 말하는 "홈페이지에서 설정한다"는 건 실제로 무엇을 뜻하나

오픈클로를 쓰다 보면 "홈페이지에서 설정했다"는 표현을 자주 쓰게 된다. 그런데 실제로는 그 말이 두 가지를 섞어 말하는 경우가 많다.

### 6-1. 브라우저 Control UI에서 바꾸는 것

문서 기준으로 Control UI에서는 이런 것들을 볼 수 있다.

- 채널 상태 확인
- 세션 확인
- 모델 오버라이드
- 크론 작업 관리
- 스킬 상태 확인
- config 수정

즉 브라우저 화면에서 꽤 많은 운영 작업을 할 수 있다. 로컬에서 열 때는 보통 아래 주소를 쓴다.

- `http://127.0.0.1:18789/`

또는 아래 명령으로 열어도 된다.

```bash
openclaw dashboard
```

실무적으로는 이 브라우저 화면을 "홈페이지"처럼 쓰게 된다. 채널이 붙었는지 보고, 설정값을 확인하고, 필요하면 config를 건드릴 수 있기 때문이다.

### 6-2. 진짜 성격과 규칙을 바꾸는 것

그런데 비서의 말투, 역할, 행동 기준 같은 건 단순히 웹 UI 버튼 몇 개로만 끝나지 않는다. 문서 기준으로도 tailoring의 핵심은 `~/.openclaw/workspace`와 `~/.openclaw/openclaw.json` 바깥 저장소 구조에 있다.

즉 브라우저 화면은 운영 패널에 가깝고, 더 깊은 성격 조정은 보통 아래 쪽에서 관리하게 된다.

- `~/.openclaw/openclaw.json`
- `~/.openclaw/workspace`

실제로 오픈클로를 오래 쓰려면 이 둘의 역할을 나눠서 이해하는 편이 좋다.

- 웹 UI: 상태 확인, 연결, config 점검
- workspace 파일: 비서의 규칙, 기억, 역할, 습관

그래서 "명령어를 설정한다"는 말도 두 층으로 나뉜다.

1. 브라우저에서 model, channel, config를 바꾸는 것
2. 워크스페이스 파일로 비서의 동작 원칙을 잡는 것

둘 중 하나만 해서는 원하는 결과가 안 나오는 경우가 많다.

## 7. 텔레그램 연결은 생각보다 단순하지만, 포인트를 모르면 계속 헷갈린다

오픈클로를 텔레그램으로 붙일 때 핵심은 세 가지다.

1. BotFather에서 토큰 만들기
2. 오픈클로 config에 텔레그램 설정 넣기
3. 게이트웨이 실행 후 pairing 승인하기

문서 기준 Quick setup 흐름은 이렇다.

### 7-1. BotFather에서 봇 만들기

텔레그램에서 `@BotFather`를 찾아서 아래 과정을 진행한다.

- `/newbot`
- 봇 이름 입력
- 유저네임 입력
- 토큰 발급받기

이 토큰이 텔레그램 연결의 시작점이다.

### 7-2. 오픈클로 설정에 토큰 넣기

문서 예시는 대략 이런 형태다.

```json5
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } }
    }
  }
}
```

여기서 중요한 포인트가 하나 있다. **텔레그램은 `openclaw channels login telegram` 같은 방식으로 붙이는 채널이 아니다.** 문서에서도 이 점을 분명히 적고 있다. 토큰을 config나 env로 넣고, 그다음 게이트웨이를 띄우는 구조다.

즉 텔레그램은 QR 로그인형 채널이 아니라 **봇 토큰 기반 연결**이라고 이해하면 된다.

### 7-3. 게이트웨이 실행 후 첫 DM 승인

```bash
openclaw gateway
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

처음 DM이 들어오면 pairing 코드가 생기고, 이걸 승인해야 DM 연결이 열린다. 문서 기준으로 pairing 코드는 1시간 안에 만료된다.

이 단계가 중요한 이유는, 텔레그램 연결이 단순히 봇 토큰만 넣는다고 끝나지 않기 때문이다. 실제로는 **누가 이 봇과 대화할 수 있는지**까지 같이 열어줘야 한다.

## 8. 텔레그램에서 자주 놓치는 설정: privacy mode와 그룹 동작

개인 DM만 쓸 거면 여기서 끝나도 된다. 하지만 그룹까지 붙이고 싶으면 BotFather의 privacy mode를 이해해야 한다.

문서 기준으로 텔레그램 봇은 기본적으로 Privacy Mode가 켜져 있어서, 그룹의 모든 메시지를 그대로 다 받지 못할 수 있다. 그룹에서 항상 반응하게 하고 싶다면 보통 아래 둘 중 하나가 필요하다.

- `/setprivacy`로 privacy mode 조정
- 봇을 그룹 관리자로 올리기

그리고 그룹에서 동작하게 만들 때는 단순히 "봇이 들어가 있다"만으로 끝나지 않는다. 오픈클로 쪽 그룹 정책도 맞춰야 한다. 이 부분을 빼먹으면 봇은 보이는데 실제로는 말을 못 받거나, 특정 사용자만 허용되는 상태가 되기 쉽다.

## 9. 결국 세팅의 핵심은 기술보다 흐름을 먼저 이해하는 것이다

오픈클로를 처음 붙일 때 자꾸 막히는 이유는 명령어가 너무 많아서가 아니다. 오히려 **각 요소의 역할이 뒤섞여 보이기 때문**이다.

- 홈페이지는 어디인가
- 내가 접속하는 화면은 무엇인가
- 모델은 API 키 경로인가, OAuth 경로인가
- 텔레그램은 QR 로그인인가, 봇 토큰인가
- 설정은 브라우저에서 끝나는가, workspace 파일로 내려가야 하는가

이 다섯 가지가 정리되면 설치 자체는 훨씬 단순해진다.

그래서 처음 시작할 때는 기능을 많이 건드리기보다 아래 순서대로 보는 편이 좋다.

1. 공식 홈페이지와 문서 주소 구분
2. 로컬 Control UI 주소 확인
3. `openclaw onboard`로 기본 구조 잡기
4. 모델 경로를 `openai`와 `openai-codex` 중 하나로 명확히 선택
5. 텔레그램은 BotFather 토큰 기반으로 연결
6. 브라우저 설정과 workspace 설정의 역할을 구분

## 실수하지 않으려고 남기는 메모

- `openclaw.ai`와 `127.0.0.1:18789`는 같은 화면이 아니다.
- 텔레그램은 `channels login`보다 bot token 설정 개념에 가깝다.
- 코덱스를 쓴다고 해서 `openai/*` API 경로와 섞으면 헷갈리기 쉽다.
- 개인용 시작이라면 OAuth 기반 코덱스 경로가 진입 장벽이 낮을 수 있다.
- 브라우저 UI는 운영 패널이고, 깊은 동작 규칙은 workspace 쪽에서 관리하는 게 낫다.
- 채널 연결은 단순 접속보다 권한과 pairing까지 봐야 끝난다.

## 마무리

오픈클로를 처음 세팅할 때 중요한 건 복잡한 명령어를 많이 외우는 일이 아니다. 더 중요한 건 **어떤 주소가 어떤 역할을 하는지, 모델 인증은 어떤 경로를 타는지, 채널 연결은 어디서 시작하고 어디서 승인되는지**를 먼저 구분하는 일이다.

그 기준만 잡히면 오픈클로는 생각보다 구조가 분명한 도구다. 공식 홈페이지는 소개를 보는 곳이고, 문서는 설명을 찾는 곳이며, 실제로 내가 만지는 화면은 로컬 Control UI다. 모델은 여러 선택지가 있지만, 개인 시작 기준에서는 ChatGPT/Codex 계정 로그인 기반 경로가 훨씬 편하게 느껴질 수 있다. 그리고 텔레그램 연결은 BotFather 토큰, 게이트웨이 설정, pairing 승인이라는 세 단계로 이해하면 훨씬 덜 헷갈린다.

한 줄로 정리하면 이렇다.
**오픈클로를 처음 붙일 때는 설치 명령어보다 먼저 주소, 모델 경로, 채널 연결 방식을 구분해서 이해해야 세팅이 빠르고 덜 헷갈린다.**

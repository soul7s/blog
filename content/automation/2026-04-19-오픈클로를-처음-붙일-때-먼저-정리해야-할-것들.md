오픈클로(OpenClaw)를 처음 보면 많은 사람이 설치 명령어부터 찾는다. 틀린 출발은 아니다. 그런데 실제로는 그 전에 먼저 이해해두는 편이 훨씬 중요한 게 있다. **이 도구가 정확히 무엇을 만드는 구조인지**를 먼저 잡지 않으면, 공식 홈페이지, 문서, 로컬 웹 화면, 텔레그램 연결, 모델 인증이 한꺼번에 섞여서 꽤 헷갈리기 때문이다.

특히 오픈클로는 그냥 웹사이트 하나 띄우는 도구가 아니다. 더 정확히 말하면 **메신저에서 부를 수 있는 개인 AI 비서를 내 환경에 붙이는 구조**에 가깝다. 그래서 설치 자체보다 먼저, 어디서 보고 어디서 설정하고 어떤 경로로 연결되는지를 구분하는 편이 훨씬 쉽다. 이번 글은 그 흐름을 기준으로 정리한 기록이다.

## 1. 오픈클로는 무엇을 “활용”하는 도구인가

오픈클로를 설치 도구처럼만 보면 자꾸 중간에서 막힌다. 실제 핵심은 설치가 아니라 **게이트웨이 하나를 중심으로 AI 모델과 채널을 연결해서, 내가 쓰는 메신저에서 비서를 부를 수 있게 만드는 것**이다.

공식 설명도 이 방향이다. 하나의 게이트웨이를 실행해두고, 여기에 모델을 붙이고, 텔레그램 같은 채널을 연결하고, 세션과 메모리, 툴 사용을 관리한다. 그래서 오픈클로를 활용한다는 말은 결국 아래 셋을 한 구조로 묶는다는 뜻에 가깝다.

1. 게이트웨이 실행
2. 모델 인증과 기본 모델 선택
3. 메신저 채널 연결

이 흐름을 먼저 이해하면 왜 오픈클로가 단순한 챗봇 앱이 아니라는 것도 자연스럽게 이해된다. 핵심은 브라우저 UI가 아니라, **항상 켜져 있는 연결 구조**에 있다.

## 2. 처음에 가장 많이 헷갈리는 건 세 가지 주소다

오픈클로를 처음 만질 때 자꾸 헷갈리는 이유는 "홈페이지"라는 말이 서로 다른 세 화면을 한 번에 가리키기 때문이다. 실제로는 역할이 다 다르다.

### 2-1. 공식 홈페이지

- `https://openclaw.ai`

이곳은 오픈클로가 어떤 프로젝트인지, 어떤 방향의 제품인지, 전체 분위기가 어떤지를 보는 입구다. 소개 페이지에 가깝다.

### 2-2. 공식 문서

- `https://docs.openclaw.ai`

실제 설정법, 채널 연결, OAuth, Telegram, 보안, Control UI 같은 설명은 대부분 여기 있다. 오픈클로를 쓰다 막히면 결국 돌아오게 되는 곳도 여기다.

### 2-3. 내가 실제로 만지는 로컬 Control UI

- `http://127.0.0.1:18789/`
- `http://localhost:18789/`

이게 많은 사람이 체감상 "오픈클로 홈페이지"라고 부르는 화면이다. 하지만 실제로는 공식 사이트가 아니라, **내가 띄운 게이트웨이의 로컬 관리 화면**이다. 문서에서는 이걸 Control UI라고 부른다.

이 셋을 구분하면 흐름이 한 번에 정리된다.

- `openclaw.ai`는 소개 페이지
- `docs.openclaw.ai`는 설명서
- `127.0.0.1:18789`는 내가 실제로 쓰는 관리 화면

처음에는 전부 비슷한 웹페이지처럼 보이지만, 역할은 완전히 다르다.

## 3. 활용 관점에서 보면 왜 `openclaw onboard`가 출발점이 되는가

오픈클로를 처음 쓸 때 수동 설정으로 하나씩 쪼개 들어가면 금방 복잡해진다. 그래서 공식 문서도 onboarding 흐름을 권장한다.

가장 짧게 보면 보통 이렇게 시작한다.

```bash
npm install -g openclaw@latest
openclaw onboard --install-daemon
```

이 onboarding은 단순 설치가 아니라 아래를 한꺼번에 잡아준다.

- 모델/인증 방식 선택
- 워크스페이스 위치 설정
- 게이트웨이 포트와 인증 설정
- 채널 연결
- 필요시 데몬 설치

즉 오픈클로는 "프로그램 하나 설치"보다, **내 환경에서 돌아갈 AI 비서 구조를 세팅하는 과정**에 더 가깝다. onboarding이 필요한 이유도 여기에 있다.

## 4. 모델이 여러 개인데도 이번에는 왜 코덱스 경로가 편했는가

오픈클로는 한 모델만 쓰는 도구가 아니다. 여러 제공자와 모델을 붙일 수 있고, 같은 OpenAI 안에서도 경로가 나뉜다. 이 지점을 처음부터 구분하지 않으면 설정이 꽤 헷갈린다.

문서 기준으로 OpenAI 쪽은 크게 두 흐름으로 나뉜다.

### 4-1. `openai/*`

- OpenAI API 키 기반
- 사용량 과금형
- 개발자 API를 직접 쓰는 구조

### 4-2. `openai-codex/*`

- ChatGPT/Codex OAuth 로그인 기반
- 계정 로그인으로 시작하는 구조
- 개인 테스트 진입이 상대적으로 편한 편

그래서 이번 설정에서 코덱스를 골랐다는 말은 단순히 "코덱스가 더 좋아 보여서"라기보다, **OpenAI API 키를 따로 발급하고 과금형 경로로 가기보다, 이미 있는 계정 기반 로그인 경로가 훨씬 빠르고 편했기 때문**에 더 가깝다.

처음 시작하는 입장에서는 이 차이가 꽤 크다. API 키 발급, 사용량 관리, 과금 감각까지 동시에 챙기기보다, 기존 계정으로 먼저 붙여보는 쪽이 심리적 진입 장벽이 확실히 낮기 때문이다.

## 5. 코덱스 기준으로 보면 설정 흐름은 이렇게 단순해진다

코덱스 경로를 선택하면 보통 아래 흐름으로 이해하면 된다.

```bash
openclaw onboard --auth-choice openai-codex
```

또는 인증만 먼저 잡고 싶다면 아래처럼 갈 수도 있다.

```bash
openclaw models auth login --provider openai-codex
```

그다음 기본 모델을 이렇게 잡는다.

```bash
openclaw config set agents.defaults.model.primary openai-codex/gpt-5.4
```

여기서 중요한 건 `openai/gpt-5.4`와 `openai-codex/gpt-5.4`를 같은 걸로 보면 안 된다는 점이다.

- `openai/...`는 API 키 경로
- `openai-codex/...`는 ChatGPT/Codex OAuth 경로

이 둘을 섞지 않는 것만으로도 설정 실수가 꽤 줄어든다.

## 6. 사람들이 말하는 “홈페이지에서 설정한다”는 말의 실제 의미

오픈클로를 쓰는 사람들끼리 이야기하다 보면 "홈페이지에서 설정하면 된다"는 말을 자주 듣게 된다. 그런데 실제로는 이 말이 두 층을 섞어서 쓰는 경우가 많다.

### 6-1. 브라우저 Control UI에서 하는 운영 설정

Control UI는 로컬 브라우저에서 여는 운영 패널에 가깝다. 문서 기준으로 이 화면에서는 이런 것들을 다룰 수 있다.

- 채널 상태 확인
- 세션 확인
- 모델 오버라이드
- 크론 작업 관리
- 스킬 상태 확인
- config 수정

실무적으로는 이 화면을 가장 자주 열게 된다. 채널이 잘 붙었는지 보고, 현재 모델 상태를 보고, 필요하면 설정을 확인하기 때문이다. 그래서 많은 사람이 이 화면을 체감상 "오픈클로 홈페이지"처럼 느끼게 된다.

### 6-2. 더 깊은 동작 원칙은 workspace와 config에서 잡는 것

하지만 비서의 말투, 역할, 기억, 운영 규칙 같은 건 브라우저 화면 버튼 몇 개로 끝나지 않는다. 이런 깊은 조정은 결국 아래 두 곳으로 내려간다.

- `~/.openclaw/openclaw.json`
- `~/.openclaw/workspace`

즉 브라우저 UI는 운영 패널이고, 실제 비서의 성격과 동작 기준은 workspace 쪽에서 길러진다. 이 차이를 모르면 설정은 했는데 "왜 내가 기대한 비서처럼 안 움직이지?"라는 느낌을 받기 쉽다.

## 7. 텔레그램 연결은 단순하지만, 핵심 포인트를 놓치면 계속 헷갈린다

오픈클로를 텔레그램으로 붙일 때 핵심은 사실 세 단계다.

1. BotFather에서 봇 토큰 만들기
2. 오픈클로 설정에 넣기
3. 첫 DM pairing 승인하기

### 7-1. BotFather에서 토큰 만들기

텔레그램에서 `@BotFather`를 찾아 `/newbot`을 실행하고, 이름과 유저네임을 정하면 봇 토큰이 발급된다. 이게 출발점이다.

### 7-2. 오픈클로 설정에 토큰 넣기

문서 예시는 대략 이런 흐름이다.

```json5
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } }
    }
  }
}
```

여기서 중요한 건 텔레그램이 `openclaw channels login telegram` 같은 QR 로그인형 채널이 아니라는 점이다. 텔레그램은 **봇 토큰 기반 연결**이라고 이해하는 게 맞다.

### 7-3. 게이트웨이 실행 후 첫 DM pairing 승인

```bash
openclaw gateway
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

처음 DM이 들어오면 pairing 코드가 생기고, 이걸 승인해야 실제로 대화가 열린다. 즉 봇을 만드는 것만으로는 끝나지 않고, **누가 이 봇과 대화할 수 있는지까지 여는 과정**이 붙어 있다.

## 8. 텔레그램을 붙이고 나면 실제 활용은 이렇게 흘러간다

설정을 끝내고 나면 그다음부터는 왜 이 도구를 쓰는지가 조금 더 명확해진다. 오픈클로의 진짜 장점은 설치 순간보다, **붙여놓고 계속 쓰는 흐름**에서 더 크게 느껴진다.

예를 들면 이런 식이다.

### 8-1. 텔레그램에서 바로 부르는 개인 비서

브라우저를 열지 않아도 텔레그램에서 바로 메시지를 보내고, 필요한 답이나 초안을 받는다. 이 지점부터 오픈클로는 단순한 설치형 툴이 아니라 **항상 연결된 개인 비서**처럼 느껴진다.

### 8-2. 브라우저 Control UI를 운영판처럼 쓰기

필요할 때는 Control UI에서 채널 상태, 세션, 크론, 스킬, config를 확인한다. 그러면 메신저는 실제 호출 창구가 되고, 브라우저는 관리 콘솔이 된다.

### 8-3. 채널과 모델을 하나의 흐름으로 묶기

모델을 바꾸거나, 기본 모델을 조정하거나, 채널 동작을 확인하는 과정이 하나의 게이트웨이 안에서 이어진다. 이 구조를 한 번 이해하고 나면 오픈클로는 기능이 많아서 복잡한 도구라기보다, **역할이 분명한 운영 구조**처럼 보이기 시작한다.

## 9. 오픈클로가 특히 잘 맞는 사람

이 도구는 모든 사람에게 똑같이 맞지는 않는다. 대신 잘 맞는 사람은 꽤 분명하다.

- 메신저로 AI 비서를 부르고 싶은 사람
- 브라우저 채팅보다 항상 연결된 구조를 더 선호하는 사람
- 여러 채널과 모델을 한 흐름으로 묶고 싶은 사람
- 내 규칙과 워크스페이스를 직접 관리하고 싶은 사람

반대로 "그냥 웹에서 한 번 질문하고 끝" 정도만 원한다면 오픈클로는 조금 과할 수 있다. 이 도구의 강점은 한 번 쓰고 마는 질문보다, **연결해두고 계속 운영하는 경험** 쪽에 있기 때문이다.

## 실수하지 않으려고 남기는 메모

- `openclaw.ai`와 `127.0.0.1:18789`는 같은 화면이 아니다.
- 텔레그램은 QR 로그인형보다 bot token 설정형 채널에 가깝다.
- `openai/*`와 `openai-codex/*`를 섞으면 설정이 헷갈리기 쉽다.
- 개인 시작 기준에서는 계정 로그인형 코덱스 경로가 더 편하게 느껴질 수 있다.
- 브라우저 UI는 운영 패널이고, 더 깊은 비서 규칙은 workspace 쪽에서 관리하는 게 낫다.
- 오픈클로의 진짜 활용은 설치 순간보다, 연결 후 계속 쓰는 흐름에서 드러난다.

## 마무리

오픈클로를 처음 붙일 때 중요한 건 복잡한 명령어를 외우는 일이 아니다. 더 중요한 건 **어떤 주소가 어떤 역할을 하는지, 모델 인증은 어떤 경로로 가는지, 텔레그램 연결은 어디서 시작하고 어디서 승인되는지, 그리고 붙인 뒤에는 실제로 어떻게 쓰게 되는지**를 먼저 구분하는 일이다.

그 기준만 잡히면 오픈클로는 생각보다 훨씬 명확한 도구다. 공식 홈페이지는 소개를 보는 곳이고, 문서는 설명을 찾는 곳이며, 실제로 내가 만지는 화면은 로컬 Control UI다. 모델은 여러 선택지가 있지만, 개인 시작 기준에서는 코덱스 OAuth 경로가 훨씬 편하게 느껴질 수 있다. 그리고 텔레그램 연결까지 끝나고 나면, 오픈클로는 단순한 설치형 프로그램보다 **메신저에서 바로 부를 수 있는 내 비서 시스템**에 훨씬 가까워진다.

한 줄로 정리하면 이렇다.
**오픈클로 활용법의 핵심은 설치 명령어를 많이 아는 것이 아니라, 주소, 모델 경로, 채널 연결, 그리고 붙인 뒤의 운영 흐름까지 한 구조로 이해하는 데 있다.**

오픈클로를 처음 세팅하려고 하면,
설치 명령어보다 먼저 헷갈리는 게 있습니다.

어디가 공식 홈페이지인지,
어디가 문서인지,
내가 실제로 들어가서 설정을 만지는 화면은 어디인지가 생각보다 잘 안 나뉩니다.

여기에 텔레그램 연결까지 붙기 시작하면,
이건 홈페이지에서 하는 건지,
터미널에서 하는 건지,
봇파더에서 하는 건지 한 번에 섞여버리기 쉽습니다.

그래서 오픈클로는 설치보다 먼저 전체 흐름을 이해하는 편이 훨씬 쉽습니다.
오늘은 그 흐름 기준으로 정리해보겠습니다.

먼저 주소부터 나눠서 보면 좋습니다.

공식 홈페이지는 아래입니다.
https://openclaw.ai

이곳은 프로젝트 소개를 보는 곳에 가깝습니다.
처음 오픈클로가 어떤 도구인지 감을 잡을 때 보는 입구라고 생각하면 됩니다.

공식 문서는 아래입니다.
https://docs.openclaw.ai

실제 설정 방법,
채널 연결,
텔레그램,
OAuth,
Control UI 같은 설명은 대부분 여기서 보게 됩니다.
막히기 시작하면 결국 가장 많이 보게 되는 곳도 여기입니다.

그리고 많은 사람들이 체감상 "홈페이지"라고 부르는 화면은 따로 있습니다.

http://127.0.0.1:18789/
또는
http://localhost:18789/

이건 공식 사이트가 아니라,
내가 띄운 오픈클로 게이트웨이의 로컬 관리 화면입니다.
문서에서는 이걸 Control UI라고 부릅니다.

즉 정리하면,

공식 사이트는 소개 페이지,
문서 사이트는 설명서,
127.0.0.1:18789는 내가 실제로 만지는 로컬 관리 화면입니다.

이 구분만 먼저 잡아도 훨씬 덜 헷갈립니다.

오픈클로는 단순히 웹페이지 하나를 띄우는 도구라기보다,
메신저와 AI 에이전트를 이어주는 게이트웨이에 가깝습니다.

게이트웨이를 띄워놓고,
여기에 모델을 붙이고,
텔레그램 같은 채널을 연결해서,
결국 메신저에서 AI 비서를 부르는 흐름을 만드는 구조입니다.

그래서 처음 세팅할 때는 보통 세 가지를 같이 보게 됩니다.

1. 게이트웨이 실행
2. 모델 인증과 기본 모델 선택
3. 텔레그램 같은 채널 연결

이걸 한 번에 잡아주는 가장 쉬운 시작점은 onboarding입니다.

예를 들면 이렇게 시작할 수 있습니다.

npm install -g openclaw@latest
openclaw onboard --install-daemon

이 흐름 안에서 모델,
워크스페이스,
게이트웨이,
채널 같은 것들을 큰 틀로 한 번에 정리할 수 있습니다.

처음부터 전부 수동으로 나눠서 잡는 것보다 이쪽이 훨씬 편합니다.

모델 이야기를 해보면,
오픈클로는 선택지가 꽤 다양합니다.
그런데 이번 설정에서 코덱스 쪽을 고른 이유는 분명했습니다.

핵심은 OpenAI API 키를 따로 발급받아 과금형으로 시작하는 방식보다,
ChatGPT/Codex 계정 로그인 기반 경로가 개인용 시작에 더 편하게 느껴졌다는 점입니다.

오픈클로 문서 기준으로 OpenAI 쪽은 크게 두 길로 나뉩니다.

하나는 `openai/*` 입니다.
이건 OpenAI API 키 기반 경로입니다.

다른 하나는 `openai-codex/*` 입니다.
이건 ChatGPT/Codex OAuth 로그인 기반 경로입니다.

그래서 정확히 말하면,
코덱스가 무조건 더 좋다기보다,
이번 시작에서는 별도의 API 키를 따로 준비하는 것보다 계정 로그인 기반 경로가 훨씬 간단해서 이쪽이 더 편했습니다.

개인적으로 처음 붙일 때는 이 차이가 큽니다.
API 키 발급,
사용량 관리,
과금 감각까지 같이 챙기기보다,
이미 있는 계정으로 먼저 시작하는 쪽이 진입 장벽이 낮기 때문입니다.

코덱스 기준으로 보면 흐름은 대략 이렇습니다.

openclaw onboard --auth-choice openai-codex

또는 로그인만 먼저 할 수도 있습니다.

openclaw models auth login --provider openai-codex

그 다음 기본 모델은 예를 들면 이렇게 잡을 수 있습니다.

openclaw config set agents.defaults.model.primary openai-codex/gpt-5.4

여기서 중요한 건
`openai/gpt-5.4`와
`openai-codex/gpt-5.4`를 같은 걸로 보면 안 된다는 점입니다.

앞은 API 키 경로,
뒤는 Codex OAuth 경로입니다.

이제 많은 분들이 말하는 "홈페이지에서 설정한다"는 표현을 다시 보면,
실제로는 대부분 로컬 Control UI를 뜻합니다.

이 화면은 브라우저에서 열고,
필요하면 아래 명령으로도 접근할 수 있습니다.

openclaw dashboard

여기 들어가면 채널 상태,
세션,
모델 설정,
크론,
스킬,
config 같은 걸 볼 수 있습니다.

그래서 체감상 홈페이지처럼 느껴지는 겁니다.
하지만 실제 성격은 소개 페이지가 아니라 운영 패널에 더 가깝습니다.

반면 비서의 말투,
역할,
기억,
운영 규칙 같은 깊은 부분은
`~/.openclaw/openclaw.json`
과
`~/.openclaw/workspace`
쪽에서 관리하게 됩니다.

즉 브라우저 화면은 운영용,
workspace는 성격과 규칙용이라고 나눠서 이해하는 편이 좋습니다.

이제 텔레그램 연결 이야기를 해보겠습니다.
오픈클로를 텔레그램으로 붙일 때 핵심은 세 단계입니다.

첫째,
BotFather에서 토큰을 만듭니다.

텔레그램에서
`@BotFather`
를 찾아서
`/newbot`
을 실행하고,
이름과 유저네임을 정하면 봇 토큰이 발급됩니다.

둘째,
이 토큰을 오픈클로 설정에 넣습니다.

문서 예시는 대략 이런 흐름입니다.

channels.telegram.enabled = true
channels.telegram.botToken = "123:abc"
channels.telegram.dmPolicy = "pairing"

여기서 중요한 포인트가 하나 있습니다.
텔레그램은 QR 로그인형 채널처럼 붙는 게 아닙니다.
즉 `channels login telegram` 방식으로 이해하면 헷갈립니다.

텔레그램은 봇 토큰 기반 연결이라고 보면 됩니다.

셋째,
게이트웨이를 실행하고 첫 DM pairing을 승인합니다.

openclaw gateway
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>

즉 텔레그램 연결은 봇 생성만으로 끝나지 않습니다.
실제로는 누가 이 봇과 대화할 수 있는지 승인하는 단계까지 가야 제대로 붙었다고 볼 수 있습니다.

그룹까지 쓸 생각이라면 하나 더 알아둘 게 있습니다.
텔레그램 봇은 기본적으로 Privacy Mode가 켜져 있어서 그룹 메시지를 다 보지 못할 수 있습니다.

이럴 때는 BotFather에서
`/setprivacy`
를 조정하거나,
봇을 그룹 관리자로 올려야 할 수 있습니다.

이 부분을 모르고 시작하면,
봇은 들어왔는데 왜 말을 못 알아듣지 같은 상황이 자주 생깁니다.

오픈클로를 붙이고 나면 실제 활용은 더 분명해집니다.
핵심은 설치 그 자체보다,
붙여놓고 계속 쓰는 흐름에 있습니다.

예를 들면,
브라우저를 열지 않아도 텔레그램에서 바로 메시지를 보내고,
필요한 초안이나 답을 받을 수 있습니다.

채널 상태나 모델 설정이 궁금하면 브라우저 Control UI에서 확인하면 됩니다.
그러면 메신저는 실제 호출 창구가 되고,
브라우저는 운영 콘솔이 됩니다.

이 구조를 한 번 이해하면 오픈클로는 기능이 많은 복잡한 도구라기보다,
메신저에서 바로 부를 수 있는 개인 AI 비서를 만드는 시스템처럼 느껴집니다.

정리하면,
오픈클로를 처음 붙일 때 중요한 건 복잡한 명령어를 많이 외우는 일이 아닙니다.

어떤 주소가 어떤 역할을 하는지,
모델 인증은 어떤 경로로 가는지,
텔레그램 연결은 어디서 시작하고 어디서 승인되는지,
그리고 붙인 뒤에는 실제로 어떻게 쓰게 되는지를 먼저 이해하는 것이 더 중요합니다.

한 줄로 정리하면,
오픈클로 활용법의 핵심은 설치 명령어를 많이 아는 것이 아니라 주소, 모델 경로, 채널 연결, 그리고 붙인 뒤 운영 흐름을 한 구조로 이해하는 데 있습니다.

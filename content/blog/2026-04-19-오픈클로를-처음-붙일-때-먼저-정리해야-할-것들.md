오픈클로를 처음 세팅하려고 하면 의외로 설치 명령어보다 먼저 헷갈리는 게 있습니다.
바로 어디가 공식 홈페이지인지, 어디가 문서인지, 그리고 내가 실제로 들어가서 설정을 만지는 화면은 어디인지입니다.

저도 처음에는 이 셋이 다 비슷한 "홈페이지"처럼 느껴졌습니다.
그런데 이걸 구분하고 나니 전체 흐름이 훨씬 쉬워졌습니다.

오늘은 오픈클로를 처음 붙일 때 헷갈리기 쉬운 부분을 한 번에 정리해보려고 합니다.
공식 홈페이지 주소, 실제 설정 화면 주소, 코덱스를 고른 이유, 텔레그램 연결 방법까지 같이 적어보겠습니다.

먼저 주소부터 정리하면 좋습니다.

공식 홈페이지는 `https://openclaw.ai` 입니다.
프로젝트 소개나 전체 분위기를 보려면 여기로 들어가면 됩니다.

공식 문서는 `https://docs.openclaw.ai` 입니다.
설치 방법, 채널 연결, OAuth, 텔레그램 같은 실제 설정 내용은 대부분 여기에서 확인하게 됩니다.

그리고 많은 분들이 "홈페이지"라고 부르는 실제 관리 화면은 따로 있습니다.
기본 주소는 `http://127.0.0.1:18789/` 또는 `http://localhost:18789/` 입니다.

이 화면은 공식 사이트가 아니라, 내가 띄운 오픈클로 게이트웨이의 로컬 웹 화면입니다.
문서에서는 이걸 Control UI라고 설명합니다.

즉 정리하면 이렇습니다.

공식 사이트는 소개 페이지이고,
문서 사이트는 설명서이고,
`127.0.0.1:18789`는 내가 실제로 접속해서 설정을 만지는 로컬 관리 화면입니다.

이 차이를 먼저 알아두면 시작할 때 훨씬 덜 헷갈립니다.

오픈클로는 단순히 웹페이지 하나를 만드는 도구라기보다, 메신저와 AI 에이전트를 이어주는 게이트웨이에 가깝습니다.
게이트웨이를 띄워놓고, 여기에 모델을 붙이고, 텔레그램 같은 채널을 연결하는 구조입니다.

그래서 처음 세팅할 때는 보통 세 가지를 같이 보게 됩니다.

1. 게이트웨이 실행
2. 모델 인증과 기본 모델 선택
3. 텔레그램 같은 채널 연결

설치 자체는 공식 문서 기준으로 `openclaw onboard` 흐름으로 보는 게 가장 쉽습니다.

예를 들면 이런 식입니다.

```bash
npm install -g openclaw@latest
openclaw onboard --install-daemon
```

이 onboarding 과정에서 모델, 워크스페이스, 게이트웨이, 채널 같은 것들을 한 번에 큰 틀로 잡을 수 있습니다.
처음부터 전부 수동으로 나눠서 잡는 것보다 이 흐름이 훨씬 편합니다.

그다음 모델 이야기를 해보면, 오픈클로는 모델 선택지가 꽤 다양합니다.
그런데 이번 설정에서는 코덱스 쪽을 고른 이유가 분명했습니다.

핵심은 API 키를 새로 파서 바로 과금형으로 시작하는 방식보다, ChatGPT/Codex 계정 로그인 기반 경로가 개인용 시작에 더 편하게 느껴졌다는 점입니다.

오픈클로 문서에서는 OpenAI 쪽을 크게 두 길로 나눠서 설명합니다.

하나는 `openai/*` 입니다.
이건 OpenAI API 키 기반 경로입니다.

다른 하나는 `openai-codex/*` 입니다.
이건 ChatGPT/Codex OAuth 로그인 기반 경로입니다.

그래서 정확히 말하면 "코덱스는 API가 아니라서 골랐다"라기보다,
이번 세팅에서는 **별도의 API 키를 따로 준비해서 usage-based billing으로 시작하기보다, 계정 로그인 기반의 코덱스 경로가 더 간단해서 선택했다**고 보는 쪽이 더 맞습니다.

개인적으로 처음 시작할 때는 이 차이가 꽤 크게 느껴집니다.
API 키를 발급하고 별도 과금 감각까지 같이 챙기기보다, 이미 있는 계정으로 로그인해 시작하는 쪽이 심리적으로도 진입이 쉽기 때문입니다.

코덱스 기준으로 기본 모델을 잡을 때는 보통 이런 흐름으로 갑니다.

```bash
openclaw onboard --auth-choice openai-codex
```

또는 로그인만 먼저 하고 싶으면

```bash
openclaw models auth login --provider openai-codex
```

그 다음 기본 모델은 예를 들어 이런 식으로 잡을 수 있습니다.

```bash
openclaw config set agents.defaults.model.primary openai-codex/gpt-5.4
```

여기서 중요한 건 `openai/gpt-5.4`와 `openai-codex/gpt-5.4`를 같은 것으로 생각하면 안 된다는 점입니다.
앞은 API 키 경로이고, 뒤는 Codex OAuth 경로입니다.

이제 많은 분들이 궁금해하는 "홈페이지에서 설정하는 방법" 이야기를 해보겠습니다.
실제로는 여기서 말하는 홈페이지가 대부분 로컬 Control UI를 뜻합니다.

이 화면은 브라우저에서 열고, 보통 아래 주소를 씁니다.

- `http://127.0.0.1:18789/`

또는 아래 명령으로 열 수 있습니다.

```bash
openclaw dashboard
```

여기 들어가면 채널 상태, 세션, 모델 설정, config 같은 것들을 확인할 수 있습니다.
그래서 체감상 "홈페이지에서 설정한다"는 말이 나오게 됩니다.

다만 여기서 하나 알아둘 점이 있습니다.
브라우저 화면에서 할 수 있는 설정과, 비서의 성격이나 규칙을 잡는 설정은 완전히 같은 층위가 아닙니다.

브라우저 화면은 운영 패널에 가깝습니다.
상태를 보고, 연결을 확인하고, config를 점검하는 곳입니다.

반면 더 깊은 규칙이나 동작 원칙은 보통
`~/.openclaw/openclaw.json` 과 `~/.openclaw/workspace` 쪽에서 관리하게 됩니다.

즉 브라우저 UI는 운영용,
workspace는 성격과 규칙용이라고 이해하면 편합니다.

이제 텔레그램 연결 이야기를 해보겠습니다.
오픈클로를 텔레그램으로 붙일 때 핵심은 세 단계입니다.

첫째, BotFather에서 토큰을 만듭니다.
텔레그램에서 `@BotFather`를 찾아 들어간 뒤 `/newbot`을 실행하고, 이름과 유저네임을 정하면 봇 토큰이 발급됩니다.

둘째, 이 토큰을 오픈클로 설정에 넣습니다.
문서 예시는 대략 이런 형태입니다.

```json5
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing"
    }
  }
}
```

여기서 중요한 포인트가 하나 있습니다.
텔레그램은 QR 로그인형 채널처럼 `openclaw channels login telegram`을 치는 방식이 아닙니다.
문서에서도 텔레그램은 토큰을 config나 env에 넣고 게이트웨이를 시작하는 방식이라고 설명합니다.

셋째, 게이트웨이를 실행하고 첫 DM pairing을 승인합니다.

```bash
openclaw gateway
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

즉 텔레그램 연결은 봇 생성만으로 끝나지 않습니다.
실제로는 누가 이 봇과 대화할 수 있는지 승인하는 단계까지 가야 제대로 붙었다고 볼 수 있습니다.

그룹까지 쓸 생각이라면 하나 더 알아둘 게 있습니다.
텔레그램 봇은 기본적으로 Privacy Mode가 켜져 있어서 그룹 메시지를 다 보지 못할 수 있습니다.
이럴 때는 BotFather에서 `/setprivacy`를 조정하거나, 봇을 그룹 관리자로 올려야 할 수 있습니다.

이 부분을 모르고 시작하면 "봇은 들어왔는데 왜 말을 못 알아듣지?" 같은 상황이 자주 생깁니다.

정리하면 오픈클로를 처음 붙일 때 중요한 건 복잡한 명령어를 많이 아는 것보다, 전체 구조를 먼저 이해하는 것입니다.

공식 홈페이지는 소개 페이지이고,
문서 사이트는 설명서이고,
로컬 Control UI는 내가 실제로 만지는 화면입니다.

모델은 여러 선택지가 있지만, 개인용 시작 기준에서는 코덱스 OAuth 경로가 훨씬 편하게 느껴질 수 있습니다.
그리고 텔레그램은 QR 로그인 채널이 아니라 BotFather 토큰 기반 연결이라고 이해하면 흐름이 훨씬 단순해집니다.

한 줄로 정리하면,
오픈클로를 처음 세팅할 때는 설치 명령어만 보는 것보다 홈페이지 주소, 로컬 관리 화면, 모델 경로, 텔레그램 연결 방식을 먼저 구분해서 이해하는 것이 훨씬 중요합니다.

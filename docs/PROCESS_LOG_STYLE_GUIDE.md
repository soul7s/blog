# Process Log Style Guide

이 문서는 `AI와 GitHub, Vercel로 이 블로그를 만든 순서` 글의 톤과 구조를 다른 사람도 재현할 수 있게 정리한 지침입니다.

## 이 스타일의 목적

이 스타일은 "잘 쓴 에세이"보다 "다음에 다시 볼 수 있는 작업 기록"에 가깝습니다.

- 감상보다 복기
- 추상적 설명보다 실제 순서
- 결과 자랑보다 판단 기준
- 긴 서사보다 바로 따라갈 수 있는 흐름

## 글의 기본 인상

읽는 사람은 이 글을 보고 아래를 기대해야 합니다.

1. 무엇을 했는지 빠르게 파악할 수 있다.
2. 왜 그 순서로 했는지 이해할 수 있다.
3. 다음에 같은 작업을 할 때 실수를 줄일 수 있다.

## 말투 규칙

- 한 문단은 1~3문장으로 짧게 쓴다.
- 한 문장에서 한 주장만 말한다.
- 과장된 표현보다 실제 경험과 선택 이유를 쓴다.
- "좋았다"로 끝내지 말고 "왜 좋았는지"를 붙인다.
- "~했다" 다음에는 가능하면 "~라서 그랬다"를 붙인다.

## 구조 규칙

이 스타일은 아래 순서를 기본 뼈대로 삼는다.

1. 한 줄 블록쿼트로 글의 성격을 먼저 선언한다.
2. 참고 자료나 준비물을 짧게 적는다.
3. 전체 흐름을 번호 목록으로 먼저 보여준다.
4. 본문은 실제 순서를 따라 번호 섹션으로 쓴다.
5. 마지막에 "실수하지 않으려고 남기는 메모"를 둔다.
6. 마무리에서는 핵심 흐름을 한 번 더 짧게 정리한다.

## 잘 맞는 섹션 제목

- 참고한 자료
- 준비물
- 전체 흐름 한눈에 보기
- 1. 시작 배경
- 2. 실제로 한 일
- 3. 이 선택이 실제로 뜻하는 것
- 4. 테스트하고 수정한 부분
- 5. 적용하거나 배포한 방식
- 실수하지 않으려고 남기는 메모
- 마무리

## 이 스타일에서 중요한 것

- "무엇을 했는지"만 쓰지 말고, 그게 어떤 구조를 의미하는지 풀어쓴다.
- 결과만 적지 말고 중간에 헷갈렸던 지점을 같이 남긴다.
- 독자가 바로 따라 하기 쉬운 순서로 정리한다.
- 같은 작업을 다시 할 미래의 나를 독자로 상정한다.

## 피해야 할 것

- 감상만 길게 적는 문장
- 불필요하게 추상적인 표현
- 순서 없는 정보 나열
- 실제로 해보지 않은 팁
- 모든 문단이 너무 길어지는 구성

## 문장 예시

좋은 예시:

> 이 글은 감상보다 작업 기록에 가깝다. 다음에 같은 블로그를 다시 만들 때 중간에 헷갈리지 않으려고 순서대로 남겨둔다.

나쁜 예시:

> 이번 경험은 굉장히 흥미로웠고 여러모로 의미가 깊었기 때문에 전체 과정을 상세히 공유하려고 한다.

좋은 예시:

> 처음 결과물은 방향만 맞는 경우가 많아서, 이 수정 단계가 생각보다 중요하다.

나쁜 예시:

> 아무튼 여러 차례의 수정과 검토를 거쳐 점점 더 나은 결과물에 가까워졌다.

## 추천 사용 방식

- 사람이 직접 쓸 때는 `scripts/create-process-log.mjs`로 초안을 만든 뒤 내용을 채운다.
- 다른 AI에게 맡길 때는 이 문서를 같이 주고, 글의 실제 사실 정보도 함께 준다.
- 초안을 만든 뒤에는 마지막 체크리스트 섹션만큼은 꼭 직접 손본다.

## 빠른 체크리스트

- 블록쿼트 첫 문장이 있는가
- 전체 흐름 번호 목록이 있는가
- 각 단계에 "왜 그렇게 했는지"가 적혀 있는가
- 마지막에 실수 방지 메모가 있는가
- 마무리가 한 번 더 요약해주는가
